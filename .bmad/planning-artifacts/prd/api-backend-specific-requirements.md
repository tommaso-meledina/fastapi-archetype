# API Backend Specific Requirements

## Project-Type Overview

fastapi-archetype is a REST API backend built with Python 3.14 / FastAPI. It exposes a minimal CRUD interface (`/dummies`) as a demonstration vehicle for 12 integrated enterprise capabilities. The API serves JSON exclusively and relies on FastAPI's built-in OpenAPI/Swagger generation for documentation.

## Technical Architecture Considerations

### Endpoint Specification

| Method | Path | Description | Request Body | Response |
|---|---|---|---|---|
| GET | `/dummies` | Return all Dummy records | None | `List[Dummy]` |
| POST | `/dummies` | Create a new Dummy record | `Dummy` (validated) | Created `Dummy` |

### Centralized Constants & Messages

All static strings, resource names, error codes, and messages SHALL be defined in central files rather than scattered through the codebase:

- **`api_constants.py`** (or similar): hosts constants for resource paths, configuration keys, and other shared values. Resource definitions MAY be organized as objects, each defining a separate REST resource (e.g., path, name, description), if the pattern proves worthwhile.
- **Error codes and messages**: defined in a single central location, each error uniquely matched to a code + message pair.

### Error Response Format

Structured error responses with:
- Unique error codes centrally defined alongside their messages
- Consistent format across all endpoints
- Leverages FastAPI's built-in 422 validation error handling, extended with application-specific error codes

### Data Schemas

- JSON request/response format exclusively
- camelCase field naming in all JSON request and response payloads (via Pydantic alias generation)
- Pydantic/SQLModel-based validation on all inputs
- Single model definition (SQLModel) serving both ORM and API validation

### API Documentation

- Automatic OpenAPI 3.x specification generated by FastAPI
- Swagger UI at `/docs`
- ReDoc at `/redoc`

## Implementation Considerations

- **Rate limiting:** deferred to Phase 2
- **API versioning:** deferred to Phase 2
- **Authentication:** external IdP integration pattern (no local token endpoint), with `AUTH_TYPE=none` for local/dev and `AUTH_TYPE=entra` for remote bearer-token validation
- **Custom Prometheus metric example:** deferred to Phase 2 (FR23a)
- **SDK generation:** not applicable
